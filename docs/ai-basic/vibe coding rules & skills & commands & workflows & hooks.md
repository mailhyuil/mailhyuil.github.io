# ai vibe coding rules & skills & commands & workflows & hooks

## rules

* 코드 작성 규칙을 명확히 정의하고 따르도록 유도

### 1) 기본 원칙

* **추측 금지**: 확신 없으면 “모름/가정”을 명시하고 코드에 TODO 남기기
* **변경 최소화**: 요구사항과 무관한 리팩토링/정리 금지
* **한 번에 크게 말고**: 큰 변경은 단계로 쪼개서 PR 단위로
* **복사-붙여넣기 금지**: 중복 로직은 함수/모듈로 추출
* **보안/개인정보 최우선**: 로그에 PII/토큰/비번 절대 금지(마스킹/삭제)

### 2) 코드 스타일/품질

* 함수는 **한 가지 책임**만 (SRP)
* 조건문 깊이 3 이상이면 **리팩토링 신호**
* 에러는 **명시적으로**: 삼키지 말고, 코드/메시지/원인 남기기
* “매직 넘버” 금지: 상수화
* 주석은 “무엇” 말고 “왜”만

### 3) 아키텍처/레이어(예: NestJS 기준)

* Controller: 요청/응답 변환만
* Service: 비즈니스 로직
* Repository/DAO: DB I/O만
* 트랜잭션 경계는 Service 레벨에서 통제
* 외부 API 호출은 Adapter/Client로 분리

### 4) 로깅/추적

* 모든 로그에 **traceId/requestId** 포함
* 에러 로그는 stack + 원인 포함, 단 **민감정보 마스킹**
* 응답에 `x-trace-id`(또는 `x-request-id`) 헤더로 내려주기

## skills

“반복 작업을 빠르고 일관되게” 하는 레시피 모음 (팀 전용 매크로 느낌)

### 핵심 스킬 목록

1. **Feature Slice 생성**

* 입력: 기능명, 엔드포인트, 데이터 모델
* 출력: controller/service/dto/repo/test/route 문서까지 한 세트

1. **Bug Fix 패턴**

* 재현 단계 정리 → 원인 후보 3개 → 로그/테스트 추가 → 최소 수정 → 회귀 테스트

1. **Refactor 안전 루틴**

* 변경 전 테스트 추가 → 작은 단위로 추출 → 타입/린트 통과 → 성능/쿼리 영향 체크

1. **Error Handling 통일**

* 에러 코드 enum/매핑
* 공통 ExceptionFilter
* 응답 포맷 통일 `{ code, message, data?, traceId }`

1. **DB 쿼리/성능 점검**

* N+1 탐지 포인트
* 인덱스 후보 추출
* 느린 쿼리 로깅 + 샘플링

### 스킬 예시(짧게)

* “이 엔드포인트 추가해줘” → **DTO validation 포함**, **테스트 기본 2개**, **에러코드 부여**, **traceId 로그**까지 자동 포함

## commands

AI에게 던지는 “고정 명령어” 세트. (짧고 강하게)

### 개발 명령어 템플릿

* `/plan` : 요구사항을 작업 단위로 쪼개고 영향 파일 리스트업
* `/implement` : 계획대로 구현(규칙 위반 금지)
* `/diff` : 변경점 요약 + 위험도(회귀 가능성) 체크
* `/test` : 어떤 테스트를 추가/수정해야 하는지 제안 + 코드까지
* `/fix` : 실패 로그/에러 메시지 기반으로 최소 수정
* `/refactor-safe` : 테스트 먼저 추가하고 리팩토링 진행
* `/perf` : 병목 후보, 쿼리/캐시/비동기 개선안
* `/security` : 로그/입력검증/권한/비밀키 노출 점검

### PR/리뷰 명령어

* `/pr-summary` : PR 설명(배경/변경/검증/리스크/롤백)
* `/review` : 코드리뷰(버그 가능성, 엣지케이스, 컨벤션 위반)
* `/changelog` : 사용자 영향 변경점만 뽑기

## workflows

“어떻게 일할지”의 표준 루트. (AI가 이 순서로 움직이게)

### 1) 기능 추가 워크플로우

1. 요구사항 요약 + 비기능(성능/보안/로깅) 체크
2. API/데이터 모델 확인
3. 구현 계획(파일/함수 단위)
4. DTO/스키마 먼저
5. Service 로직
6. Controller 연결
7. 테스트(유닛/통합 최소 2개)
8. 로깅(traceId) + 에러코드 정리
9. `/diff`로 변경점/리스크 요약

### 2) 버그 수정 워크플로우

1. 재현 조건 정리(입력/상태/환경)
2. 가설 2~3개 + 로그 포인트 제안
3. 실패 테스트 먼저 작성
4. 최소 수정
5. 회귀 테스트 추가
6. 롤백/모니터링 포인트 메모

### 3) 리팩토링 워크플로우

1. 목표(가독성/성능/중복 제거) 명시
2. 보호 테스트 추가
3. 작은 단위로 추출/교체
4. 타입/린트/테스트 통과
5. 성능/쿼리 변화 체크
6. 변경점 문서화

## hooks

“자동으로 실행되는 안전장치” (AI가 실수해도 잡아주는 애들)

### 권장 훅(로컬)

* **pre-commit**

  * format + lint + typecheck (빠르게)
  * 금지어 스캔(토큰/키/비번 패턴)
* **pre-push**

  * unit test
  * 변경 범위가 크면 integration test 일부

### 권장 훅(명령 실행 후)

* AI가 코드 생성/수정했을 때 자동:

  * import 정리
  * formatter 적용
  * 타입 에러 있으면 “에러 원인+수정안”을 바로 제시

### 권장 훅(CI)

* PR 열리면:

  * lint/typecheck/test
  * 커버리지 임계치(예: 70% 미만이면 경고)
  * 마이그레이션/스키마 불일치 체크

### 훅 예시(행동 규칙)

* 테스트 실패하면:

  * “테스트를 지우지 말고” 코드 고치기
* 린트 실패하면:

  * 규칙 무시 옵션 추가 금지(진짜 필요하면 근거를 남기고 최소 범위로)
